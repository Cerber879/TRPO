**Дата:** 20 сентября 2025 г.

**Статус:** Принято

**Контекст:**

Нужно выбрать процесс разработки, который обеспечит координацию работы команды, контроль качества кода, управление версиями и планирование релизов.

**Рассмотренные варианты:**

1. **Git Flow:** Классическая модель ветвления с выделенными ветками для разработки, релизов и исправлений. Подходит для проектов с регулярными релизами.
2. **GitHub Flow:** Упрощенная модель с основной веткой и feature-ветками. Подходит для проектов с частыми деплоями.
3. **Trunk-Based Development:** Разработка в одной основной ветке с короткоживущими feature-ветками. Требует высокой дисциплины команды.
4. **Scrum/Kanban:** Методологии управления проектами с итеративной разработкой и регулярными ретроспективами.

**Решение:**

Выбран упрощенный подход на основе GitHub Flow с элементами Scrum для планирования итераций.

**Обоснование:**

- **Простота:** GitHub Flow проще в освоении и поддержке, чем Git Flow, что снижает когнитивную нагрузку на команду.
- **Гибкость:** Позволяет быстро вносить изменения и деплоить их без сложной координации веток.
- **Интеграция:** Хорошо интегрируется с GitHub и его инструментами (Pull Requests, Issues), которые мы уже активно используем.
- **Планирование:** Scrum-подход обеспечивает структурированное планирование спринтов и контроль прогресса.
- **Опыт команды:** У нас уже есть опыт работы с GitHub Flow и Scrum, что позволит быстро внедрить процесс без лишних обсуждений.

**Недостатки решения:**

- **Ограниченная изоляция:** Меньше изоляции между различными этапами разработки по сравнению с Git Flow.
- **Требования к дисциплине:** Требует ответственного подхода к code review и тестированию перед merge.

**Последствия:**

- Основная ветка (main/master) должна всегда находиться в рабочем состоянии.
- Каждая новая функциональность разрабатывается в отдельной feature-ветке.
- Перед merge в основную ветку обязателен code review через Pull Request.
- Рекомендуется использовать Issues для отслеживания задач и планирования спринтов.
- Необходимо настроить базовые проверки (linting, тесты) перед merge.
